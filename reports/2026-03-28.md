**Likely cause**: The check script is building a comma-separated import list and producing a leading `(,` when the list starts with a separator. This is a recurring false-positive artifact in the check script itself, not a real code issue. The pipeline `outcomes` map records this as `"typecheck": "success"`, indicating this check is treated as informational/non-blocking.

**Suggested fix**: Review `check-versions.sh` or the template import check logic to ensure the generated import statement doesn't have a leading comma.

- `# Per-turn token usage (added v0.1.50)` — API documentation comment
- `# Skill names to load (added v0.1.50)` etc. — field-level version annotations
- `**Fix** (v0.1.51): ... On v0.1.50, override manually:` — workaround note
- `**Workaround** (v0.1.50 only): ...` (×4) — active workaround notes for Known Issues
- Changelog entry `| v0.1.50 | ... |`

**Likely cause**: The verify script's stale-version regex matches all occurrences of the old version string, including intentional historical references (changelog rows, "added in vX.Y.Z" API annotations, "fixed in vX, workaround on vX-1" notes). These must not be replaced. The mending agent is also unable to fix this because the referenced lines are correct documentation.

**Suggested fix**: The verify script needs an allowlist/exemption for patterns like `v${oldVersion}+`, `(added v${oldVersion})`, changelog table rows, and `(v${oldVersion} only)` workaround headers. This is a structural issue in the verify logic, not in the skill content.
